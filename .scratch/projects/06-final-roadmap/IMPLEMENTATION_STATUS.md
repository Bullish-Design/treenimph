## What's NOT Yet Implemented

### From the Initial Concept

| Feature | Concept Section | Priority | Notes |
|---|---|---|---|
| **README.md stub generation** | "Scope of TreeNimph-Generated Files" | Low | Concept says export should generate a `README.md` stub. Not implemented. |
| **Round-trip integration tests** | Concept Review "Testing Strategy" | Medium | Tests that export → `tree-sitter generate` → parse sample source → verify parse tree. Current `test_examples.nim` validates + renders but doesn't run `tree-sitter generate` on the output. |
| **Comparison tests** | Concept Review | Low | Hand-written `grammar.js` compared against TreeNimph output. The snapshot test partially covers this. |
| **Error message snapshot tests** | Concept Review | Low | Validation errors are tested by content matching, not full snapshot comparison. |
| **Source-location-aware validation** | Concept — "Optional Higher-Level Checks" | Post-v1 | Explicitly deferred in concept. Current approach (grammar-domain locations) is the correct v1 strategy per the concept review. |

### From the Query Concept

The query concept outlines a 5-phase rollout. Current status:

| Phase | Status | What It Means |
|---|---|---|
| **Phase 1**: Stub files | **Done** | Export writes empty query stubs |
| **Phase 2**: Raw string/file passthrough | **Done** | `QueryFiles` type with `Option[string]` for each category, exported properly |
| **Phase 3**: Class-based query IR | **Not started** | `NodePattern`, `FieldPattern`, `Capture`, `Predicate`, etc. |
| **Phase 4**: Category-specific models | **Not started** | `HighlightRule`, `TagDefinitionRule`, `LocalScopeRule`, etc. |
| **Phase 5**: Refined authoring story | **Not started** | Path A vs Path B decision, cross-validation against grammar |

### Higher-Level Concept Goals Not Yet Addressed

1. **Larger real-world grammar example** — The concept review recommended a 20-30 rule grammar to honestly stress-test verbosity. The largest example (json_like) has 10 rules. A grammar like a simplified Python, Lua, or TOML would be a better proof point.

2. **Documentation** — The concept mentions docs and the README is minimal. No doc comments on public procs (noted in the code review). This is expected for MVP but needed before wider adoption.

3. **Package publishing readiness** — The nimble file is set up, but no CI, no published package, no versioning strategy beyond `0.1.0`.

4. **"Suspicious self-references" and "accidental unreachability" checks** — Listed as optional higher-level validation. The unreferenced-rule warning is a start, but self-referencing cycles (rule that only refers to itself) and truly unreachable rules (not reachable from the entry point) are not detected.

5. **Rule graph visualization** and **documentation generation from grammar objects** — Listed as future extensions. Not started.

6. **Richer helper library** — The concept envisions growing the helper set based on real usage. Current set (4 helpers) is intentionally minimal per the concept review's recommendation.

## Summary

**The MVP fully implements the concept's Phases 1-5** (Core Model → Validation → Rendering → Export → Toolchain Integration) **and most of Phase 6** (Polish). Every "Must Address" item from the concept review has been resolved. The query rollout is at Phase 2 of 5 (passthrough support).

**Remaining work to reach the full concept vision:**

| Priority | Item |
|---|---|
| **High** | Larger (20-30 rule) example grammar to validate ergonomics at scale |
| **High** | Round-trip integration test (export → `tree-sitter generate` → parse) |
| **Medium** | README.md stub generation on export |
| **Medium** | Doc comments on all public procs |
| **Medium** | Unreachable-from-entry-point rule detection |
| **Low** | Query DSL Phase 3+ (class-based query IR) |
| **Low** | Rule graph visualization |
| **Low** | Property-based tests for renderer invariants |
| **Low** | CI / package publishing infrastructure |

The library is in a strong position — the core architecture is sound, all concept review recommendations were adopted, and the remaining items are either polish, testing depth, or explicitly deferred future features.
